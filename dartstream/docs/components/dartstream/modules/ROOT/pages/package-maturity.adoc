= Package Maturity
:page-layout: default

DartStream packages should be labeled by maturity so developers can understand
what is ready for normal use and what is still a reference or experiment.

== Labels

[cols="1,3",options="header"]
|===
| Label
| Meaning

| Ready
| Recommended for normal application use. The package has a stable contract, useful docs, and passing core tests.

| Preview
| Usable for early adopters, but needs stronger docs, tests, or provider validation before default recommendation.

| Experimental
| Evaluation-only package or example. API and behavior may change.

| Internal
| Not intended for public use or pub.dev publication.
|===

== Current Direction

* Standard engine and base contracts should move toward Ready first.
* Provider packages should not be called Ready until their real provider flows are validated.
* Experimental examples must stay clearly labeled.
* Server-side frameworks that compete with DartStream should not be documented as preferred integrations.

== Initial Classification

[cols="1,1",options="header"]
|===
| Package or Area
| Label

| `ds_dartstream_standard_engine`
| Preview

| `ds_dartstream_standard_engine_extension`
| Preview

| `ds_auth_base`, `ds_database_base`, `ds_storage_base`, `ds_feature_flags_base`, `ds_message_broker_base`, `ds_websocket_base`
| Preview

| `ds_ai_base`, `ds_orm_base`
| Experimental

| `ds_intellitoggle_provider`
| Preview

| `ds_flagd_provider`
| Experimental

| Auth, database, storage, logging, and reactive provider packages
| Preview or Experimental until provider-specific QA confirms readiness

| Example frontend folders
| Experimental unless the example has current setup docs and validation
|===
